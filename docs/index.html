这个 skill 是什么

在已有成熟前端项目里实现设计稿时，强制 AI 先比对、再复用、最后才新写，并把每次用户的修正沉淀为项目级规则，复用准确率随使用提升。

解决的问题：成熟项目都有完整组件库，但 AI 直接看图写代码会写出和组件库脱节的 <div> 堆。这个 skill 通过 6 步纪律把 AI 拉回"先 import 现成组件"的轨道。

触发条件

会触发

  • "按这个设计稿做一个页面" / "还原这个设计"
  • "把 Pencil / Figma / 这张图写成代码"
  • 上传图片/截图/.epgz 并要求落地到当前项目

不会触发

  • 新项目脚手架（无组件库可复用）
  • 纯设计评审（"这个布局好不好看" → impeccable）
  • 仅抽 token 不写组件（→ figma-token-extractor）

6 步工作流

① 询问扫描策略
→
② 解析设计稿
→
③ 比对匹配
→
④ 用户确认
→
⑤ 生成代码
→
⑥ 学习沉淀
1

项目首次接入 — 询问扫描策略

检查 <repo>/.comate/component-inventory.md 是否存在；不存在时必须先问：

  • A. 按需扫描（推荐）：边做边查，当前设计稿涉及的组件类型才索引，增量缓存
  • B. 全量预扫：一次扫遍组件库，慢但后续匹配更稳
2

解析设计稿

按输入类型分流（详见 输入解析），输出 modules[]：每个模块带 id、语义标签、视觉特征、子模块。

3

组件复用比对（最关键）

按下文 匹配优先级 顺序查找，每个模块产出 REUSE / EXTEND / NEW 三类决策之一。

4

匹配确认（必须等用户确认）

把比对结果以表格呈现：

模块决策组件置信度备注
顶部导航REUSEsrc/components/AppHeader0.95完全匹配
用户卡片EXTENDCard + 新 variant=profile0.70需加头像槽
折线图NEW——库内无
未确认前不要写代码。若用户调整决策，立即写入映射库（见 Step 6）。
5

生成代码

  • REUSE 直接 import，禁止复制粘贴组件内部样式
  • EXTEND 改组件源码加 prop/variant，遵循原有命名风格
  • NEW 用项目 design tokens，禁止硬编码魔法值

全程匹配项目代码风格（先读 2-3 个邻近文件再写）。完成后简述："复用 N 个组件、扩展 M 个、新增 K 个文件"。

6

学习沉淀（让 skill 越用越准）

触发条件（任一即触发）

  • 用户在 Step 4 调整了匹配决策
  • 用户改了 AI 写的代码，diff 涉及组件 import 替换（如 <div> → <UserCard>）
  • 用户明确说"以后这种设计直接用 XX 组件"

写入位置

  • 项目独有规则 → <repo>/.comate/component-mapping.md
  • 跨项目通用启发 → ~/.comate/skills/design-to-code-reuse/references/heuristics.md

写入后告诉用户："已学到规则 #N：xxx，写入 [文件路径]，下次同类设计会自动命中。"

匹配优先级

  1. 项目映射库 .comate/component-mapping.md — 用户历史修正沉淀的硬规则，命中即用
  2. 项目清单 .comate/component-inventory.md — 按语义 + 视觉特征模糊匹配
  3. Explore 子代理增量扫 — 按需扫描模式下未命中时检索（如 candidate_role=card 时搜 Card|Tile|Panel）
  4. 跨项目启发式 references/heuristics.md
  5. 仍未命中 → NEW

三类决策

REUSE 直接复用
置信度 ≥ 0.8
import 现成组件，列出 props 映射。最理想状态。
EXTEND 复用并扩展
置信度 0.5 ~ 0.8
复用但需新增 variant / prop（如 Button 缺 tone="warning"），改组件源码后使用。
NEW 新写
无候选
必须新写，遵循 design tokens 与项目代码风格。同类设计反复出现时考虑沉淀为新组件。

- path: `src/components/Button/index.tsx`
- exports: `Button` (default), `ButtonProps`
- variants: primary | secondary | ghost | danger
- sizes: sm | md | lg
- key_props: { onClick, disabled, loading, icon, tone }
- visual_signature:
  - 圆角: 6px (token: --radius-md)
  - 高度: 32/40/48 按 size
  - 主色: token --color-brand
- semantic_tags: [button, cta, action]

映射规则 schema

位置：<repo>/.comate/component-mapping.md · skill 的长期记忆

## Rule 001 — 用户头像 + 名称卡片用 UserCard

- 触发条件：模块包含「圆形头像（左）+ 姓名（粗体）+ 副标题（灰色）」
  组合，且整块有浅色背景或边框
- 决策：使用 `src/components/UserCard`，props 映射：
  - 头像 → avatarUrl
  - 姓名 → name
  - 副标题 → subtitle
- 不适用：仅头像无文字（用 Avatar）；横向列表里的简版（用 UserChip）
- 来源：2026-06-17 用户修正 — "这种应该用 UserCard 不是自己拼 div"

写入纪律

  • 新规则编号递增，不复用已删除编号
  • 触发条件要可观测（视觉特征、语义角色），不写"看起来像 XX"这种主观描述
  • 决策必须给出 import 路径与 props 映射
  • 必填"不适用"字段；缺反例时写 TBD（待用户后续反馈补充）
  • 来源必须有日期 + 简述上下文

输入解析

1. 图片 / 截图

vision 自顶向下分块（page → 区块 → 子模块），抽取 bounding box、形状、文字层级、状态指示。

2. Pencil 文件

优先 Pencil MCP 拿结构化节点（节点名常含语义如 card-user）；次选导出 PNG 走图片分支。

3. Figma

本 skill 不直接访问 Figma；引导用户先用 figma 或 figma-token-extractor，再回到本 skill 第 3 步。

4. candidate_role 受限词汇

解析时只能用以下标签，避免瞎猜：

navigationsidebarheaderfootercardlistlist-itemtableformform-fieldbuttonicon-buttonbadgetagavatarmodaldrawerpopovertabsbreadcrumbpaginationchartempty-statebannerunknown

关键原则

1. 先读后写 — 动手前必须完成 Step 3 比对 + Step 4 确认。跳过比对直接写代码是这个 skill 最大的反模式。
2. 复用优于完美还原 — 5% 视觉差异（如圆角差 2px）优先用现有组件并告知用户，除非明确要求"严格按设计稿"。
3. 学习是默认行为 — 每次用户修正都尝试写入映射库；让用户 review 规则是否合理，而不是等用户主动说"记下来"。
4. 不污染代码库 — .comate/ 工作产物提示用户纳入版本控制以便团队共享。
