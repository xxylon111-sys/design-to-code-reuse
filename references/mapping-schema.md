# Component Mapping Schema

`<repo>/.comate/component-mapping.md` 是项目级"设计特征 → 组件"映射规则库。每条规则来源于用户的真实修正，是 skill 的长期记忆。

## 文件头

```yaml
---
project: <repo-name>
updated: 2026-06-17
rule_count: 0
---
```

## 规则格式

```markdown
## Rule 001 — 用户头像 + 名称卡片用 UserCard

- **触发条件**：设计模块包含「圆形头像（左）+ 姓名（粗体） + 副标题（灰色）」组合，且整块有浅色背景或边框
- **决策**：使用 `src/components/UserCard`，props 映射：
  - 头像 → `avatarUrl`
  - 姓名 → `name`
  - 副标题 → `subtitle`
  - 整体可点击 → `onClick`
- **不适用**：仅头像无文字（用 `Avatar`）；横向列表里的简版（用 `UserChip`）
- **来源**：2026-06-17 用户修正 — "这种应该用 UserCard 不是自己拼 div"
- **置信度阈值**：模块满足全部三要素时直接采用；缺一项时降级为建议
```

## 写入规则（skill 自动追加时遵循）

1. **新规则编号递增**，不复用已删除编号
2. **触发条件要可观测**：写视觉特征（形状、布局、文字层级）或语义角色，不要写"看起来像 XX"这种主观描述
3. **决策必须给出 import 路径与 props 映射**，模糊描述无效
4. **必填"不适用"字段**：避免规则被滥用到错的场景；如果当时没有反例信息，写 `TBD（待用户后续反馈补充）`
5. **来源必须有日期 + 简述上下文**，便于将来判断规则是否过时
6. **写入后通知用户一句话**："已学到规则 #N：[标题]，写入 .comate/component-mapping.md"

## 规则冲突处理

匹配时如果多条规则都命中：
- 优先取**触发条件更具体**（约束更多）的那条
- 仍并列时，优先取**最近添加**的那条（用户偏好可能演进）
- 把冲突情况告知用户，让用户决定是否合并/废弃旧规则

## 规则失效

每次匹配前快速校验规则中引用的组件路径是否存在；不存在则标记 `status: stale`，匹配时跳过并提示用户更新。
